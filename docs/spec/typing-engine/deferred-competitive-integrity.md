## 位置づけ

MVP の方針：

- **タイマーはクライアント側で素直に管理** する（`requestAnimationFrame` + `performance.now()`）。
- 不正対策は **`paste` 無効化** と **サーバー側のスコア上限チェック**（120 秒で 1500 文字超のスコアを HTTP 400 で拒否）の 2 点のみ。
- 入力環境（Bluetooth キーボード遅延、低スペック端末）への補正や警告は行わない。

この方針で **「寿司打レベルのカジュアル競技性」** は確保される想定。本格的な競技ゲームのような厳密な公平性（クライアント時刻改ざん対策、環境差吸収など）は MVP 後の課題として保留する。

## 着手トリガー

以下のいずれかが発生したときに本ドキュメントの設計案を検討し、実装計画を立てる：

1. **ユーザーからの問い合わせ・苦情**
   - 「上位プレイヤーが明らかに不正している」「マクロを使っている」
   - 「タブを切り替えてカンニングする人がいる」
   - 「Bluetooth キーボードだと不利だ」
   - 「自分のスコアが低スペック PC のせいで伸びない」
2. **トップ 10 ランキングに不審なスコアが記録される事象**
   - サーバー上限（120 秒で 1500 文字）の手前ギリギリ、または人間離れした高 accuracy が連発する等
3. **コミュニティ・SNS で「不正対策が甘い」という言及が一定数発生** したとき

これらが起きない限り、MVP の素朴な実装（`paste` 無効化 + スコア上限チェックのみ）で運用を続ける。**プロアクティブな実装はしない**：問い合わせベースで動く。

## 設計案

### サーバー権威タイマー

クライアントタイマーは UX 用、**スコア計算と検証はサーバー権威の時間** で行うことで、クライアント時刻改ざんを防ぐ。

- `/start` 時にサーバーが `startAt = Date.now()` を Redis に保存。
- `/finish` 到達時に `elapsed = serverFinishAt - startAt` を計算。
- **`elapsed` が 120,000ms ± 500ms 以外なら `flagged=true`** としてランキング集計から除外。
- 許容差 500ms はネットワーク遅延と RTT を考慮した値。実装後に分布を観測してチューニング。

実装上の追加：

- `play_sessions` に `startedAt`（サーバー時刻）と `serverElapsedMs`（権威タイマー値）を追加。
- API レスポンスに `flagged` を含め、リザルト画面でユーザーに告知（不要なら表示しなくてもよい）。

### タブ非アクティブ時のセッション無効化

「タブを開いたまま裏でカンニング」「タイマーが止まっている間に答え合わせ」を防ぐ。

- クライアント側で `document.visibilitychange` を監視し、プレイ中にタブが非アクティブになったら：
  - `POST /api/play-sessions/:id/invalidate` を呼び、Redis の sessionId に `invalidated=true` フラグを立てる
  - クライアントは「タブを離れたためセッションが中断されました」と表示し、もう一度トライ可能に
- `/finish` 時に `invalidated=true` を検出したら DB に書き込まない / `persisted=false` を返す。

実装上の追加：

- `POST /api/play-sessions/:id/invalidate` API。
- Redis ステートに `invalidated(bool)` フィールド。
- `/finish` のロジック分岐追加。

### キーストローク間隔の異常検知

マクロや自動入力ツールによる人間離れした打鍵速度を検知してランキングから除外する。

- 「100 キー連続で 10ms 未満間隔」など、明らかに人間でない打鍵パターンを `keystroke_logs` から抽出。
- 統計的な閾値（中央値間隔の極端な外れ値）でも検知。
- 検知されたセッションは `flagged=true` とし、ランキング集計から除外。

実装上の追加：

- `play_sessions.flagged(bool)` カラムを追加。
- `/finish` でサーバー側にキーストロークログを受け取った時点で統計解析。
- ランキング集計バッチで `WHERE NOT flagged` を適用。

### 出題シーケンスの整合性検証

クライアントが「サーバーが出題していない問題のスコア」を捏造する不正を防ぐ。

- `/start` 〜 `/finish` の間、Redis に保持したサーバー側の出題シーケンスと、クライアントが `/finish` で送信する問題 ID シーケンスを比較。
- 不一致なら HTTP 400 で拒否、または `flagged=true`。

実装上の追加：

- `/finish` のリクエストボディに `problemSequence: [problemId, ...]` を含めるよう拡張。
- サーバー側で Redis シーケンスとの突き合わせ処理。

### 回帰検知用の数値ベースライン（INP / Lighthouse）

体感品質のドリフトを CI / 本番で検知する。

- **INP（Interaction to Next Paint）p95 < 50ms** を本番 RUM で監視。50ms を超え始めたらパフォーマンス劣化のシグナルとして調査。
- Lighthouse スコア（Desktop プリセット）を CI で継続的に追跡。
- 劣化を検出したら開発チームに通知（Slack / メトリクスダッシュボード）。

実装上の追加：

- フロントに `web-vitals` ライブラリ等を組み込み、INP / LCP / CLS を計測してテレメトリエンドポイントに送信。
- 専用テーブル / 外部の RUM サービス（Vercel Analytics, Datadog RUM 等）に集約。

### 入力環境の差への対応

| 観点 | 対応案 |
| --- | --- |
| **Bluetooth キーボードの遅延（数十 ms）** | 開始画面に「有線キーボード推奨」の警告を表示。`navigator.bluetooth` 等で検知できれば自動アラート |
| **低スペック端末** | INP 監視で検知し、閾値超のクライアントを **別ランキング**（例：「カジュアルランキング」）に分ける |
| **モニターのリフレッシュレート差（60Hz vs 144Hz）** | rAF ベース計測で吸収済みのため対応不要 |
